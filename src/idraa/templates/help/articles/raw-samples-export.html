What the file is

Every completed analysis run also has a downloadable raw sample export: a gzipped, wide CSV holding the actual per-iteration Monte Carlo values that produced that run’s Annualized Loss Expectancy (ALE), Value at Risk (VaR), and expected shortfall figures. Where the PDF and the verification workbook give you Idraa’s computed summaries, this export gives you the underlying numbers themselves so you can independently recompute VaR, expected shortfall, or the loss exceedance curve with your own tools.

The file has one row per simulated iteration and one column per stored array — for a 100,000-iteration run that is 100,000 rows. Above the header row, every line begins with # and carries provenance for the file: the run id, the run type, the iteration count, the random seed (and any per-scenario derived seeds), the inputs hash, the app version that generated it, and a generation timestamp. Treat that preamble as the file’s chain of custody — it is what lets you prove which run, and which exact inputs, a given CSV came from.

Find the download next to Download PDF and the Excel verification workbook on the run detail page, labelled Download raw samples (CSV). It only appears while that run’s sample arrays are still retained (see Availability below).

Column naming

The first column is always iteration, and it is 0-based — the first simulated draw is row iteration = 0, not 1. The remaining columns depend on whether the run is a single-scenario run or an aggregate (portfolio) run:

Run typeColumns
Single-scenario base_risk (inherent, before controls) and residual_risk (after controls).
Aggregate (portfolio) aggregate_without_controls and aggregate_with_controls for the portfolio total, plus one scenario_<hex>_base_risk / scenario_<hex>_residual_risk pair for every scenario in the run.

The <hex> in a per-scenario column name is that scenario’s UUID with the hyphens removed (32 lowercase hex characters) — never a scenario name, and never a hyphenated UUID. Column names have to survive being pasted into pandas, R, or a spreadsheet formula unescaped, and hyphens and punctuation in a name would break that; the hex id is the stable, copy-paste-safe machine key.

Because the hex id alone isn’t human-readable, the #-comment preamble includes a legend line for every scenario, mapping its hex id back to its scenario name and full (hyphenated) UUID — for example column scenario_<hex>_*: scenario_id=<uuid> name=Ransomware. Read the legend once and you can label every scenario_<hex>_* column pair by name.

Precision

Sample values are exported as float32 — the same precision the arrays are stored at internally. Idraa’s in-app metrics (the headline ALE, VaR, and expected-shortfall figures shown on the run page, in the PDF, and in the verification workbook) are computed from float64 at simulation time. Recomputing the same statistic from this export should agree with the in-app figure to roughly 1e-6 relative precision — close agreement, not bit-for-bit equality. A visible difference in, say, the 9th significant digit of a VaR estimate is expected float32/float64 rounding, not a data error.

Size envelope

At Idraa’s production caps — 100,000 iterations, roughly 30 scenarios in a single aggregate run — the export is on the order of 75 MB expanded and 25–35 MB compressed (the file you actually download). Smaller runs are proportionally smaller; a 10,000-iteration single-scenario run is a few hundred kilobytes. Size scales roughly linearly with iteration count and with the number of scenarios in an aggregate run, so a run with more scenarios or a higher iteration count produces a larger file, not a slower page — the export streams in chunks rather than building the whole file in memory first.

Availability

Raw samples are retained under Idraa’s sample retention policy, which is shorter than how long the run’s summary results (ALE, VaR, the charts) are kept — the sample arrays are the largest part of a run and are purged first. Once a run’s samples have been purged, the download link disappears from the run detail page, and requesting the export URL directly returns 410 Gone rather than a file.

A 410 is not an error to report — it means exactly what it says: the raw arrays for that run are gone. If you need them again, re-run the analysis with the same random seed (visible in the run’s own metadata and in this export’s preamble on any earlier copy you kept) to regenerate an equivalent set of samples. Because export and every other bulk-data download in Idraa is a sensitive action, you may also be prompted to re-authenticate with your password or passkey first if your session’s authentication isn’t recent — the same step-up check used by the PDF, the workbook, and the scenario import/export flows.

Recomputing tail metrics yourself

The preamble lines all start with #, so any CSV reader that supports a comment prefix will skip them automatically and land on the header row. Two worked examples, using the residual_risk column from a single-scenario run:

Python (pandas)

import pandas as pd
df = pd.read_csv("run-<id>-samples.csv.gz", comment="#")
var95 = df["residual_risk"].quantile(0.95)
es95 = df.loc[df["residual_risk"] >= var95, "residual_risk"].mean()

R

df <- read.csv(gzfile("run-<id>-samples.csv.gz"), comment.char = "#")
quantile(df$residual_risk, c(0.95, 0.99))

Both readers transparently decompress the .gz file, so there is no need to unzip it first. For an aggregate run, swap residual_risk for aggregate_with_controls, or for one scenario’s own column pair using the hex id from the preamble legend.
