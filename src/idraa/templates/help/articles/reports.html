The executive PDF report

Every completed analysis run includes a downloadable PDF report designed for a board, CFO, or risk committee audience. To generate one, open the run detail page at /analyses, then click Download PDF.

The PDF is rendered server-side from the stored simulation results — no browser plugin is required, and the figures and numbers are drawn from that stored snapshot rather than being recomputed on each download.

Every completed run also has a companion downloadable Excel verification workbook: an independent, in-Excel re-derivation of the same Monte Carlo math using native spreadsheet formulas, so a stakeholder can check Idraa’s numbers without trusting the app’s engine. Find it next to Download PDF on the run detail page.

Reading control attribution (Shapley values)

When multiple controls cover the same scenario, their risk-reduction contributions overlap: removing any one control changes the result by less than its standalone benefit would suggest. Simple subtraction double-counts that overlap; attribution solves it by allocating credit fairly.

Idraa uses Shapley values from cooperative game theory (Shapley, 1953; Castro et al., 2009) to split the total risk reduction across controls. Each control’s Shapley value is its average marginal contribution over every possible order in which controls could have been added to the portfolio. This satisfies four axioms: efficiency (credits sum exactly to the total reduction — no rounding gap), symmetry (controls with identical impact share credit equally), null player (a control that adds nothing receives zero), and linearity (attribution composes across independent scenarios).

Methodology note

Control attribution via Shapley values is a v3 view-model / portfolio reporting derivation — not FAIR-grounded. Cooperative game theory is not a node in the Open FAIR Standard (which defines risk through Loss Event Frequency, Loss Magnitude, and their FAIR-CAM sub-functions). Shapley attribution lives in Idraa’s view-model service layer, not in the FAIR engine, so that the FAIR math layer stays uncontaminated by aggregation policy choices. References: Shapley, L. S. (1953). “A Value for n-Person Games.” Contributions to the Theory of Games, Vol. II; Castro, J., Gómez, D., & Tejada, J. (2009). “Polynomial calculation of the Shapley value based on sampling.” Computers & Operations Research, 36(5).

How to read the attribution table

The attribution table lists each control, its Shapley value in dollars, and its share of the total modelled risk reduction as a percentage. The percentages sum to 100 % (the efficiency axiom guarantees this). A control that appears in many overlapping scenarios may have a smaller percentage than its standalone simulation would suggest — that is the correct answer, not a calculation error. Controls with very little overlap retain most of their standalone credit.

Use the attribution table to prioritise renewal and investment decisions: controls with the highest Shapley share deliver the most unique, non-redundant risk reduction across your scenario portfolio. Controls near zero may be redundant with higher-ranked peers and are candidates for rationalisation.

Computation limits

For portfolios with up to 12 controls, Idraa computes Shapley values exactly via full subset enumeration (212 = 4,096 evaluations). For larger portfolios it uses the permutation-sampling estimator from Castro et al. (2009), with a sample count derived from the Maleki (2013) finite-sample bound at relative error ε = 2 % and confidence 1 − δ = 95 %. Both paths are exactly efficient: the per-control credits always sum to the true total risk reduction.

Average, typical, and if-removed figures in the PDF

The same control-value framing used elsewhere in Idraa carries into the PDF’s attribution tables: an average (mean) Shapley figure paired with a muted typical-case figure, an “if removed” (leave-one-out) drop-cost column alongside the fair-share Shapley attribution, and — for runs computed with the weight-robustness ensemble — a robustness range per control. Downloading and sharing the PDF

The Download PDF button is available to all authenticated users (Analyst, Reviewer, Viewer, Admin). Every download is recorded in the audit log against the user who triggered it. Because downloading a report is a sensitive action, Idraa may prompt you to re-authenticate (with your password or passkey) first if your session’s authentication isn’t recent — the same step-up check applies to the data exports covered in the Import & export guide.

Because the PDF is rendered from the stored run snapshot, the figures, tables, and numbers are identical every time you re-download the same run — useful when you need to confirm that the report shared with the board matches the current system record. The cover page and footer do stamp the time you generated that particular copy, so two downloads of the same run will show different generation timestamps even though the underlying data is unchanged.
