Overview

Idraa lets you move scenarios in bulk via import and export in two formats: CSV and JSON. Export produces a file you can archive, share, or re-import into another workspace; import is the reverse — a two-step upload-then-confirm flow that validates each row before writing anything. The pair is designed for a clean round-trip: a file exported from Idraa can be re-imported without data loss.

Role requirement: export is available to any authenticated user (Analyst, Reviewer, or Admin). Import is restricted to Admin users only — the upload form and both template downloads are 403-blocked for Analyst and Reviewer roles.

Because export and import both move data in bulk, Idraa treats them as sensitive actions: if your session’s authentication isn’t recent, you may be prompted to re-authenticate with your password or passkey before the download or upload proceeds.

Exporting scenarios

Any authenticated user can export from the Scenarios list page (/scenarios). Use the Export dropdown at the top of the list to choose between CSV and JSON. The download honours whatever ?status= filter is active, so you can export only active scenarios, archived scenarios, or the full set.

You can also export a single scenario from its detail page using the same dropdown. The resulting file is named scenario-<id>.csv or scenario-<id>.json.

Both formats carry the same fields: scenario name, description, threat actor type, asset class, all FAIR input parameters (TEF, Vulnerability, Primary Loss, Secondary Loss ranges), status, and creation timestamps. Every export is recorded in the audit log.

Importing scenarios (Admin only)

Navigate to /scenarios/import. The page provides download links for a blank CSV template (scenarios_template.csv) and a JSON sample (scenarios_sample.json) so you can see the expected column layout and field names before authoring your file.

Step 1 — Upload and preview

Choose your CSV or JSON file and click Upload. Idraa validates every row without writing anything to the database. The preview page shows:

  • Rows that passed validation and are ready to import.
  • Rows that are duplicates of existing scenarios (skipped automatically).
  • Rows with validation errors, with a per-row error message.

If the file is unparseable (wrong format, truncated, or exceeds the 5 MB size limit) the upload returns an error immediately and nothing is stored. The preview is held server-side under a short-lived token (10 minutes); if the token expires before you confirm, you will see an “expired preview” message and must re-upload.

Step 2 — Confirm

Review the preview and click Confirm import. Only the rows that passed validation are written. Duplicate rows are silently skipped; error rows are also skipped and listed in the result summary. A successful import with no errors redirects you to the Scenarios list; a partial import (some rows valid, some invalid) shows a result page so you can see what was created and what was rejected.

Each imported scenario is owned by the importing Admin’s organisation and is recorded in the audit log.

Round-trip fidelity

A file exported from Idraa is designed to re-import cleanly without data loss or field drift. The export format matches the import parser’s expected column names and value encodings, so the round-trip — export → (optionally edit) → re-import — is lossless for all fields carried in the format. Scenarios re-imported this way are treated as new records; they do not overwrite the originals, which lets you use export-then-import as a duplication mechanism.

If you edit the file between export and import, only rows that pass validation will be written. Common editing mistakes — invalid enum values, missing required fields, or numbers outside the allowed range — surface as per-row errors in the preview step before anything is committed.

Format notes

CSV — each row is one scenario; the header row names the columns. Download the template to get the exact column order. Numeric FAIR parameters (e.g. TEF low / high, loss low / high) use decimal notation. String enumerations (threat actor type, asset class) must match the values shown in the template.

JSON — a JSON array of scenario objects; field names match the CSV column names. The sample file provides one worked example per common threat actor type. JSON is preferable when field values may contain commas or newlines (e.g. long descriptions).

Other ways data moves in and out

Beyond scenario CSV/JSON, every completed analysis run has its own downloadable Excel verification workbook — an independent, in-Excel re-derivation of the Monte Carlo math using native spreadsheet formulas, so a stakeholder can check Idraa’s numbers without trusting the app’s engine. Find it next to Download PDF on the run detail page; see the Reports guide for details.

If you are migrating from a qualitative risk register (a spreadsheet of High/Medium/Low ratings), Admins have a separate guided import at /register-import: a multi-step flow that maps your register’s columns and values onto FAIR inputs and previews the resulting draft scenarios before anything is created. This is distinct from the CSV/JSON scenario import above, which expects Idraa’s own native format.
